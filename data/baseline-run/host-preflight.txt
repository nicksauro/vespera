## Anomalies / observations for baseline-run validity

1. Project venv missing: `.venv/Scripts/python.exe` does not exist on host. Any baseline-run command that expects it will fail. Must be provisioned before T002.0a G09 baseline-run begins.
2. Current memory pressure at capture time is MODERATE (47.4% used): ~8.1 GB already consumed by user workload at t=0. Top consumers: vmmem (~703 MB - WSL2/Docker), Discord (~598 MB), chrome.exe (multiple instances aggregating ~1.6 GB+), claude.exe (~519 MB), MsMpEng.exe (~320 MB - Windows Defender). Baseline-run should either (a) shut down these processes first, OR (b) budget against `available=9.0 GB`, NOT `total=16.0 GB`.
3. Hardware is older-generation: Intel i7-3770 (Ivy Bridge, released 2012), DDR3-1333. Relevant for throughput assumptions in Aria's ADR-1 v2 - do NOT assume modern-CPU memory bandwidth. 4 physical cores / 8 logical.
4. PageFile is SYSTEM-MANAGED (not user-pinned): currently 40.85 GiB allocated, historical peak 31.3 GiB. A memory-intensive baseline-run that forces heavy paging could expand the pagefile further - disk I/O on C: drive becomes the failure mode, not hard OOM. R1/R2/R3 thresholds must distinguish 'RAM pressure' from 'commit pressure'.
5. Hyper-V / WSL2 active: `Hipervisor detectado` + vmmem running. RAM measurements above already reflect hypervisor overhead. Any baseline-run that spawns additional WSL/Docker workloads will compound memory pressure.
6. wmic deprecation notice: `wmic` still works on 10.0.19045 but is flagged for removal. All wmic captures above were cross-validated with PowerShell `Get-CimInstance` where possible. Future re-captures should prefer CIM cmdlets.
7. BIOS-level system info is placeholder: Manufacturer/Model = 'To be filled by O.E.M.' - this is a custom / whitebox build, no vendor PSU / thermal specs available.
