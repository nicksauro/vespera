## User story

Como Beckett rodando CPCV 45 paths, preciso de **dois** adapters plugáveis atrás da mesma interface (`Iterable[Trade]` do contrato atual de `HistoricalTradesReplay`): um `feed_timescale.py` (slow path, queries ad-hoc contra DB) e um `feed_parquet.py` (hot path, mmap sobre os parquets materializados em T002.0a), para que eu alterne entre exploração (DB) e produção (parquet) sem mudar o engine downstream, e para que contract tests provem que ambos honram o lock de hold-out.

## Acceptance criteria

### Interface compartilhada

- [ ] **AC1** Ambos adapters produzem o mesmo tipo `Trade(ts: datetime, price: float, qty: int)` já definido em `packages/t002_eod_unwind/core/session_state.py:20` (NÃO redefinir; importar e reutilizar). Rationale (Beckett 2026-04-22): spec T002 v0.2.0 usa apenas `(close, open, high, low, ATR_20d)` — aggressor/ticker não são load-bearing para CPCV. Para T003+ que precise aggressor/book, criar novo `TradeEnriched` em nova story (dado custodial preservado nos parquets T002.0a)
- [ ] **AC2** API: `load_trades(start_brt: datetime, end_brt: datetime, ticker: str) -> Iterable[Trade]` — ordenados por `ts` ascendente, sem gaps artificiais, sem dedupe silencioso (se houver dupes no source, expor flag). `ticker` é WHERE filter (whitelist `{"WDO","WIN"}`, ValueError fora). Assert runtime `trade.ts.tzinfo is None` em ambos adapters (BRT naive). Empty iterator é válido (sessão vazia em feriado parcial), não raise

### feed_timescale.py

- [ ] **AC3** Conecta via variáveis de `.env.vespera` (lidas via `os.environ`; **não** hardcode); falha explícita se `VESPERA_DB_*` ausentes
- [ ] **AC4** Query parametrizada, prepared statement: `SELECT timestamp, price, qty FROM trades WHERE ticker = %s AND timestamp >= %s AND timestamp < %s ORDER BY timestamp ASC` (aggressor fica custodial no parquet, não entra no stream do adapter per Beckett 2026-04-22)
- [ ] **AC5** Stream cursor (server-side cursor `cursor('name')` do psycopg2) — **não** carregar sessões inteiras em memória; yield trade-a-trade
- [ ] **AC6** BRT naive preservado (timestamp sem tz_convert); assert em runtime que `trade.ts.tzinfo is None`

### feed_parquet.py

- [ ] **AC7** Descobre arquivos via `data/manifest.csv`; falha explícita se manifest ausente ou se sha256 não bater (integridade)
- [ ] **AC8** mmap via `pyarrow.parquet.ParquetFile` + `iter_batches(batch_size=65536)` — não materializar o dataset inteiro
- [ ] **AC9** Filtro de janela aplicado via predicate pushdown (partitioning por year/month): só abre parquets de meses tocados pela janela
- [ ] **AC10** BRT naive (pyarrow retorna `timestamp[ns]` sem tz dado nosso schema em T002.0a)

### Hold-out lock (contract test)

- [ ] **AC11** Ambos adapters, se receberem janela intersectando `[2025-07-01, 2026-04-21]` com `VESPERA_UNLOCK_HOLDOUT != 1`, lançam `HoldoutLockError` **antes** de qualquer I/O (DB connection ou parquet open) — erro é fail-fast
- [ ] **AC12** Contract test `tests/contracts/test_adapter_holdout_lock.py`:
  - `feed_timescale.load_trades(2025-07-01, 2025-07-02, "WDO")` → `HoldoutLockError`
  - `feed_parquet.load_trades(2025-07-01, 2025-07-02, "WDO")` → `HoldoutLockError`
  - Mesmo teste com `VESPERA_UNLOCK_HOLDOUT=1` (fixture `monkeypatch.setenv`) → passa (retorna iterable, ainda que vazio)

### Performance budget (Beckett)

- [ ] **AC13** Performance budget — separar **cold start** vs **hot loop CPCV**:
  - **(a) Cold start:** primeira leitura de uma sessão WDO (~500k-850k trades) ≤ **10s** em hardware dev. Inclui abertura de arquivo parquet + deserialização de schema + primeiro iter_batch. Medido em `tests/perf/test_parquet_speed.py` via pytest-benchmark, fixture limpa (sem cache aquecido).
  - **(b) Hot loop CPCV:** replay completo de **1 path sobre ~250 sessões** ≤ **4.5 min target**, amortizando cache de sessões em memória (sessões reutilizadas entre folds do CPCV não pagam cold-start novamente). Medido em `tests/perf/test_parquet_cpcv_loop.py`.
  - **Validação Beckett:** handshake pré-aceitação com **medição real sobre 3 paths representativos** do CPCV 45-paths: `baseline` (sessão típica), `high_vol` (dia de expiry/FOMC), `low_vol` (sessão pós-feriado). Beckett assina AC13(b) com evidência numérica na story.
- [ ] **AC14** Comparação ponto-a-ponto: `feed_timescale` e `feed_parquet` para mesma janela produzem **exatamente** os mesmos trades `(ts, price, qty)` em mesma ordem (diff-hash teste em `tests/integration/test_adapter_parity.py`). Hash sobre tuplas 3-field; aggressor discard é simétrico em ambos adapters, não gera assimetria observável

### Regressão downstream

- [ ] **AC15** `HistoricalTradesReplay` existente aceita output de ambos adapters sem modificação de código do replay (comprovação: teste que pluga cada adapter → 32/32 tests T002.3 continuam passando)

- [ ] **T2** — `packages/t002_eod_unwind/adapters/feed_timescale.py` (AC3-AC6, AC11). Usa `_holdout_lock.assert_holdout_safe(start, end_inclusive)` **antes** de `psycopg2.connect()`. Query exatamente como AC4 (3 cols: timestamp, price, qty). Server-side cursor via `conn.cursor(name='ts_stream')`. Yield `Trade(ts, price, qty)` importado de `core/session_state.py`. Ticker validado via whitelist `{"WDO","WIN"}` — ValueError se outro.
- [ ] **T3** — `packages/t002_eod_unwind/adapters/feed_parquet.py` (AC7-AC10, AC11). Usa `_holdout_lock.assert_holdout_safe(start, end_inclusive)` **antes** de abrir qualquer parquet. - [ ] **T5** — `tests/integration/test_adapter_parity.py` (AC14): janela pequena `2024-03-04 → 2024-03-05` (2 sessões WDO, ~1.5M trades), assert lista de `(ts, price, qty)` byte-a-byte idêntica entre 2 adapters via hash MD5 do pickle.
