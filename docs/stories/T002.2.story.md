## User story

Como engine de sinal T002 processando trades em sequência, preciso manter estado
de sessão agregado em O(1) por trade (open_day, last_price, high/low para ATR_day,
closed_bars registry) e derivar as 3 features canônicas (`intraday_flow_direction`,
`intraday_flow_magnitude`, `atr_day_ratio`) em snapshots determinísticos nas
janelas {16:55, 17:10, 17:25, 17:40} BRT, para que backtest e live compartilhem
exatamente a mesma lógica (layers 2-3 puras do design).

## Acceptance criteria

- [ ] **AC1** `SessionState` atualiza `open_day`, `last_price`, `session_high`,
  `session_low`, `last_ts` em O(1) por trade; primeiro trade >= 09:30:00 BRT
  define open_day
- [ ] **AC2** `SessionState.snapshot_at(ts)` retorna `close_at_ts = último preço
  <= ts` (nunca futuro) — anti-leakage
- [ ] **AC3** `FeatureComputer.compute(state, entry_ts, atr_20d, pct_bands)`
  retorna `Features(intraday_flow_direction, intraday_flow_magnitude,
  atr_day_ratio)` — tudo determinístico, zero side effects
- [ ] **AC4** `intraday_flow_direction ∈ {-1, 0, 1}` (sign exato, `close == open_day
  ⇒ 0`)
- [ ] **AC5** Se ATR_20d == 0 (edge), computer retorna `magnitude = inf` e sinaliza
  flag `regime_atr_undefined` — NÃO crasha
- [ ] **AC6** BRT naive em todos timestamps (grep `tz_convert` / `utcnow` /
  `pytz.UTC` → zero)
- [ ] **AC7** Trades-only — zero dependência de book (R7)
- [ ] **AC8** Coverage ≥ 85%
- [ ] **AC9** Lint + typecheck passam
- [ ] **AC10** Testes de parity com spec YAML: dado um cenário sintético,
  computer retorna features **iguais** ao que a formula documentada em
  `docs/ml/specs/T002-...yaml` prescreve (Sable audit)
