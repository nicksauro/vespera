## User story

Como Beckett preparando o backtest T002, preciso de uma `signal_rule` pura que
traduza features + `trial_id ∈ {T1..T5}` em `Signal(LONG|SHORT|FLAT, reason)`
conforme trading_rules da spec, e de um `HistoricalTradesReplay + BacktestBroker`
que reproduz a sessão WDO trades-only e emite fills determinísticos (mesmo
modelo de slippage Roll+1tick), para que eu possa rodar CPCV 10/2/45 paths
sobre o dataset in-sample 2024-01 → 2025-06 sem jamais tocar o hold-out
2025-07 → 2026-04 (pré-registrado).

## Acceptance criteria

### Signal rule (layer 4)

- [ ] **AC1** `compute_signal(features, bands, trial_id) -> Signal` é **função
  pura**: mesma entrada ⇒ mesma saída, zero side effects, zero I/O, zero logging
- [ ] **AC2** Baseline T1 implementada exatamente conforme spec
  `trading_rules.entry_conditions`:
  - entra se `magnitude > P60` **E** `atr_day_ratio ∈ [P20, P80]`
  - `direction = -sign(flow_direction)` (FADE)
  - caso contrário `FLAT` com `reason` explicativa
- [ ] **AC3** Variants T2..T5 implementadas conforme
  `spec.n_trials.variants` (T2 P50, T3 P70, T4 sem regime filter, T5 apenas
  janela 17:25). Cada variant recebe sua própria banda P (injetada, não
  hardcoded) — signal_rule não sabe qual P aplicar, apenas consome o threshold
- [ ] **AC4** Edge cases: `flow_direction == 0` ⇒ `FLAT` com reason
  `"flow_flat"` (não pode faceletar zero); `magnitude == inf`
  (`regime_atr_undefined` da T002.2) ⇒ `FLAT` com reason `"atr_undefined"`
- [ ] **AC5** `Signal.reason` é string human-readable determinística para cada
  combinação (`"magnitude_below_P60"`, `"atr_ratio_outside_regime"`,
  `"entered_fade_short"`, `"entered_fade_long"`, `"flow_flat"`,
  `"atr_undefined"`)

### Backtest feed adapter (layer 1 histórico)

- [ ] **AC6** `HistoricalTradesReplay` consome iterável de `Trade(ts, price, qty)`
  ordenados por ts ascendente; emite `TradeEvent` idêntico ao contrato do feed
  live (mesmo schema) — zero divergência entre backtest e live no layer 2 acima
- [ ] **AC7** Dia inválido (Copom pós-filtro, feriado, rollover D-3..D-1) é
  filtrado **antes** do replay entrar no session state — usa `CalendarData.is_valid_sample_day`

### Backtest execution adapter (layer 5 backtest)

- [ ] **AC8** `BacktestBroker.execute(signal, n_contracts, ts, mid_price)`
  retorna `Fill(ts, price, qty, fees_rs)`:
  - **slippage model:** `Roll_spread_bps + 1 tick worst-case` (spec
    `costs.slippage_model` — Beckett default)
  - **fees:** brokerage + emolumentos + IR day-trade (inicialmente usa
    defaults conservadores R$0.25/contrato/lado; TODO Nova atlas)
  - LONG ⇒ paga `mid + slippage`; SHORT ⇒ recebe `mid - slippage`
- [ ] **AC9** `exit_deadline_brt = 17:55:00` é **hard stop**: se houver posição
  aberta às 17:55, broker força fill a mercado com mesmo slippage model (não
  respeita nenhum outro critério)
- [ ] **AC10** Zero lookahead: `BacktestBroker` nunca consulta ts > entry_ts

### Integração end-to-end

- [ ] **AC11** Teste de integração `test_backtest_one_day.py` — cenário
  sintético de 1 sessão: trades dummy → warm-up OK → signal dispara às 17:10 →
  broker fill → deadline 17:55 força saída → PnL calculado = `(exit - entry)
  × mult × n × side - fees`. Valores conferem com cálculo manual em pytest.
- [ ] **AC12** Teste determinismo: mesmo input ⇒ mesmo output byte-a-byte
  através de 3 execuções consecutivas (R6 manifest — CPCV precisa disso)
- [ ] **AC13** BRT naive em tudo; grep `tz_convert`/`utcnow`/`pytz.UTC` → zero
- [ ] **AC14** Coverage ≥ 85% no conjunto novo; lint + typecheck passam
