## ¬ß1 Purpose & scope

Mira Gate 4b T5 verdict (`GATE_4_FAIL_data_quality`, 2026-04-30) attributed K3 IC=0 with CI95=[0,0] to an **engineering wiring gap**, not a strategy-edge falsification. Specifically:

- `ReportConfig.ic_in_sample = 0.0` / `ic_holdout = 0.0` / `ic_spearman_ci95 = (0.0, 0.0)` defaults flow verbatim through `compute_full_report` to `MetricsResult` and `evaluate_kill_criteria` (`report.py:185-189, 527-528, 554-555`).

- `packages/vespera_metrics/info_coef.ic_spearman()` + `bootstrap_ci()` exist and are correct, but are **orphan from the dry-run path** ‚Äî no caller wires them into the report-producing chain.

This review adjudicates **WHERE** in the architecture the IC computation MUST be wired (post-fanout aggregation, pre-`compute_full_report` consumption) so that:
1. The closure-isolation invariant from T002.1.bis APPROVE_OPTION_B is preserved verbatim.
2. The D-1 anti-leak invariant from T002.6 T0b ¬ß2.4 (Mira spec ¬ß3.4) is preserved verbatim.
3. The hold-out lock (R1 + R15(d)) is UNTOUCHED.
4. Mira Gate 4b spec yaml v0.2.3 thresholds (UNMOVABLE per Anti-Article-IV Guard #4) are NOT mutated.
5. Determinism stamp (R15 + Beckett ¬ß5) absorbs IC computation provenance cleanly.

NO Mira ML semantics override (Mira owns WHAT IC measures and HOW the predictor‚Üîlabel pairing is constructed; Aria authority is purely WHERE the computation lives in the call graph and WHICH typed object owns the result propagation contract). This document binds **engineering placement only**.

## ¬ß2 Placement option A ‚Äî IC inside `compute_full_report()`

**Mechanism:** add a new parameter `cpcv_results -> ic_spearman` computation inside `compute_full_report` body (after Step 1-2 PBO matrix build, alongside Step 3 DSR).

### ¬ß2.1 Architectural objections

1. **Tight coupling of measurement + reporting concerns.** `compute_full_report` is currently a *consumer* of pre-computed primary statistics (DSR via `deflated_sharpe_ratio`, PBO via `probability_backtest_overfitting`) plus an *aggregator* of secondary metrics from the per-fold `AggregateMetrics`. IC is structurally different: it requires per-event predictor‚Üîlabel pairs (Mira spec ¬ß0.3 falsifiability ‚Äî "the falsifying statistic MUST be honestly measured against the data"), which `BacktestResult` does NOT carry today (see ¬ß5 below). Placing IC inside `compute_full_report` forces the report module to either (a) reach into `BacktestResult.trades` and reconstruct the predictor‚Üîlabel pairing ‚Äî which is **strategy-domain logic, not metrics-domain logic** ‚Äî or (b) require a NEW input parameter alongside `cpcv_results`. Both leak strategy-domain knowledge into the metrics module.

2. **`ReportConfig` semantics break.** Today `ReportConfig.ic_in_sample` is documented at `report.py:167-170` as *"Caller supplies these from upstream IC computation (NOT derived from BacktestResult, which carries trade-level data not predictor-vs-label IC)"*. Option A inverts this contract: the report module derives IC internally. The dataclass-as-frozen-input contract is broken.

3. **Test surface explosion.** Six existing tests in `tests/vespera_metrics/test_report.py` (T002.0f T2 AC5 + AC6) instantiate `compute_full_report` against synthetic `cpcv_results` fixtures with the documented invariant *"IC fields are caller-supplied; defaults render zero"*. Option A forces every fixture to either (a) carry per-event predictor/label arrays or (b) accept that internal IC computation against synthetic data may produce non-zero values, which breaks deterministic assertions on `MetricsResult.ic_spearman == 0.0` in baseline-fixture tests.

4. **Bootstrap CI seed plumbing duplicated.** `info_coef.bootstrap_ci(sample, statistic, n_resamples, confidence, seed)` already takes a `seed` parameter (Efron 1979 / PCG64). Inside `compute_full_report`, the bootstrap CI seed would have to be threaded from `ReportConfig.seed_bootstrap` into the IC computation path ‚Äî adding a second consumer of the same seed alongside future bootstrap consumers. Manageable but adds a determinism-anchor surface that the report module previously did not own.

5. **Single-Responsibility Principle (Article-IV-trace lens).** The report module's responsibility per `report.py:1-13` docstring is: *"MetricsResult / FullReport / KillDecision dataclasses + compute_full_report"* ‚Äî **assembly, not measurement**. Adding IC measurement violates the SRP envelope established at T002.0f T2 sign-off.

**Verdict:** REFUTED.

## ¬ß3 Placement option B ‚Äî IC in dry-run script between fanout and report

**Mechanism:** in `scripts/run_cpcv_dry_run.py` after `cpcv_results = run_5_trial_fanout(...)` (line 1050) and before `cfg = ReportConfig(...)` (line 1070), invoke a new `compute_ic_from_cpcv(cpcv_results, ...) -> (ic_in_sample, ic_holdout, ic_ci95)` function and pass results into `ReportConfig(seed_bootstrap=seed, ic_in_sample=..., ic_holdout=..., ic_spearman_ci95=...)`.

### ¬ß3.1 Architectural arguments

**For:**
1. **Decoupled from report module.** `compute_full_report` signature unchanged; `ReportConfig` field semantics preserved verbatim (per `report.py:167-170` docstring: *"Caller supplies these from upstream IC computation"*). T002.0f T2 AC5+AC6 tests stay green untouched.
2. **Closure isolation preserved.** IC computation runs **post-fanout** (after `runner.run` returns), in the orchestration layer, NOT inside the per-fold `backtest_fn` closure. The T002.1.bis APPROVE_OPTION_B factory pattern + closure R/O captures (`enclosed_costs / enclosed_calendar / enclosed_percentiles`) are NOT touched.
3. **Hold-out lock untouched.** The dry-run script's hold-out gate (`scripts/run_cpcv_dry_run.py:1142-1152`) fires BEFORE any iteration; IC computation lives downstream of fanout, post-data-consumption.

**Against:**
1. **Orchestration coupling for any other caller.** Any future consumer of `compute_full_report` (paper-mode T002.7, hold-out clearance T002.8, future epics) MUST replicate the IC wiring stub or accept the all-zero defaults. The wiring contract becomes a **convention**, not a structural invariant.
2. **No reusable contract.** A second dry-run script (e.g., a Beckett-only N-prime wall-time perf-round) would have to re-implement the wiring inline. **Code duplication risk.**
3. **Determinism provenance harder to anchor in DeterminismStamp.** The IC computation seed propagation (PCG64 via `bootstrap_ci(seed=...)`) lives outside `BacktestRunner.build_determinism_stamp` ‚Äî adding `ic_seed_source` to the determinism stamp requires script-level wiring, not engine-level. Manageable but an extra fork in the determinism audit trail.

**Verdict:** **VIABLE BUT NOT CANONICAL.** Option B is a working short-circuit; for a single caller (current dry-run) it is sufficient. For multi-caller extensibility (Phase G/H paper-mode + future hold-out clearance), it is fragile.

## ¬ß4 Placement option C ‚Äî IC inside CPCV engine post-fanout aggregation

**Mechanism:** extend `CPCVEngine.run` (or `BacktestRunner.run`) to also compute IC after the per-fold loop, attaching it to a new field on the return value (e.g., `(results, aggregate_metrics)` tuple, or a new `RunSummary` wrapper).

### ¬ß4.1 Architectural objections

1. **Engine responsibility drift.** `CPCVEngine` is documented at `engine.py:96-108` as *"stateless-after-construction CPCV engine"* with the responsibility envelope: partition + purge + embargo + per-fold dispatch + result ordering. Adding IC aggregation introduces **statistical-measurement** logic into the engine ‚Äî a clean SRP violation. Mira spec authority over IC computation semantics + Aria authority over CPCV partition semantics would now overlap inside the same module.

2. **Engine signature creep.** Today `engine.run` returns `list[BacktestResult]` (45 entries per call). Option C requires a wider return type or an out-parameter. Either breaks back-compat for the 4+ existing callers (`runner.py:138`, three tests in `tests/cpcv/`).

3. **Mira ML-domain ownership leaks into Beckett-domain engine.** `vespera_cpcv` is the Beckett-authored CPCV implementation per Mira spec `T002-cpcv-purge-formula.md`; IC is Mira-domain ML-statistics. Cross-domain coupling violates the agent-authority boundary established by ESC-011 R7+R8.

4. **Parallelization preservation under stress.** `CPCVEngine` reserves the right to parallelize per-fold execution (T002.1.bis Aria ¬ßI.1 + `engine.py:106-108`). IC computation is a **post-fanout reduction** that cannot run in the per-fold parallel scope; placing it inside the engine forces the engine to also own the join barrier, complicating the parallel design.

5. **Determinism stamp duplication.** `BacktestRunner.build_determinism_stamp` already records `seed`, `dataset_sha256`, `cpcv_config_sha256`. IC's bootstrap CI seed would need a separate stamp field; engine-internal aggregation makes that field engine-coupled rather than caller-injected.

**Verdict:** REFUTED.

## ¬ß5 Placement option D ‚Äî Separate orchestration module `vespera_metrics/cpcv_aggregator.py`

**Mechanism:** create a new public function (NOT a new file unless contract grows) `compute_ic_from_cpcv_results(cpcv_results, *, seed_bootstrap, predictor_label_extractor) -> ICResult` that consumes the per-fold output, applies a Mira-defined predictor‚Üîlabel extractor (per F2 spec amendment Mira will draft as F2-T0a), and returns a typed `ICResult` consumable directly by `ReportConfig`. Library lives in `packages/vespera_metrics/` alongside `info_coef.py`. Caller invokes it once in `run_cpcv_dry_run.py` between fanout and report.

### ¬ß5.1 Architectural arguments

**For:**
1. **Single-Responsibility intact.** New module owns one job: aggregate per-fold trial-event data into IC + CI95. Reuses `info_coef.ic_spearman` + `bootstrap_ci` (no duplication, kills the orphan-code finding from Mira ¬ß4.2).
2. **`ReportConfig` semantics preserved.** Caller supplies IC values via the existing fields; doc-string contract `report.py:167-170` honored verbatim. T002.0f T2 AC5+AC6 tests stay green.
3. **Closure isolation preserved.** Computation runs post-fanout in the orchestration layer; no per-fold backtest_fn touch.
4. **D-1 anti-leak preserved by construction.** The `predictor_label_extractor` callable defined per Mira F2 spec amendment OWNS the in-sample-vs-holdout discipline; Aria condition C-A1 below mandates it operates on `(cpcv_results[trial][i].fold.train_session_dates, .test_session_dates, .trades)` so the function CANNOT cross fold boundaries by signature.
5. **Determinism stamp absorbs cleanly.** The new function takes `seed_bootstrap` as parameter; the dry-run script forwards `cfg.seed_bootstrap` to both IC and PBO bootstrap consumers from a single source of truth.
6. **Reusable across callers.** Phase G/H paper-mode (T002.7) and hold-out clearance (T002.8) inherit the wiring as a one-line public API call: `ic = compute_ic_from_cpcv_results(cpcv_results, seed_bootstrap=cfg.seed_bootstrap)`. No code duplication.
7. **Article IV trace clean.** Function lives in `vespera_metrics`, calls `info_coef.ic_spearman` (orphan code ‚Üí consumed), records its own provenance fields in the `ICResult` dataclass; trace path is one-hop.
8. **Mira ML-domain authority preserved.** The `predictor_label_extractor` is a Mira-authored callable (defined in F2 amendment spec); Aria provides the engineering placement, Mira provides the semantic content. Cross-domain authority boundary respected.

**Architectural cost (honest accounting):**
- +1 public function in `packages/vespera_metrics/` (<= ~60 LoC body + ~20 LoC `ICResult` dataclass).
- +1 consumer line in `scripts/run_cpcv_dry_run.py` (replace line 1070 `cfg = ReportConfig(seed_bootstrap=seed)` with `ic = compute_ic_from_cpcv_results(cpcv_results, seed_bootstrap=seed); cfg = ReportConfig(seed_bootstrap=seed, ic_in_sample=ic.in_sample, ic_holdout=ic.holdout, ic_spearman_ci95=ic.ci95)`).
- +1 export line in `packages/vespera_metrics/__init__.py`.
- +N tests in `tests/vespera_metrics/test_cpcv_aggregator.py` (per Quinn QA F2-T2/T3).
- Existing surfaces (`info_coef.py`, `report.py`, `engine.py`, `runner.py`, `cpcv_harness.py`) UNCHANGED.

**Architectural cost-benefit:** strongest factor ‚Äî *zero touch* on T002.1.bis factory pattern + T002.6 closure body + T002.0f compute_full_report contract + CPCV engine. Modification surface is a new module + one caller line.

**Verdict:** **APPROVED.**

## ¬ß7 ReportConfig field semantics

Mira ¬ß4 audit established that the current defaults (`0.0` / `0.0` / `(0.0, 0.0)`) collapse exactly to "field not wired" and produce the IC=0 with CI95=[0,0] signature surfaced at Beckett N7. Aria recommendation:

### ¬ß7.1 KEEP defaults at zero (NOT mutate to None/sentinel)

**Rationale:**
1. The docstring at `report.py:187-189` already documents *"(0.0, 0.0) indicates 'not provided' upstream, which the report renders verbatim"* ‚Äî Article IV honest rendering, NOT invention. The defaults are HONEST signals to downstream consumers.
2. Switching to `None` defaults forces `ic_spearman: float` ‚Üí `ic_spearman: float | None` in `MetricsResult` (`report.py:42`), which cascades through `to_markdown` (line 110-113) and `_serialize_full_report` (`scripts/run_cpcv_dry_run.py:789-790`) ‚Äî wider surface.
3. The fix is at the **caller side** (Option D), not the defaults side. Once the dry-run script wires `compute_ic_from_cpcv_results`, the defaults become the "no-IC fallback" used only by tests and by future callers that legitimately do not have IC data (e.g., a single-fold debugging script). Honest semantics.

### ¬ß7.2 ALTERNATIVE not chosen ‚Äî callable IC computer in ReportConfig

A field `ic_computer: Optional[Callable[[CPCVResults], ICResult]]` would allow `compute_full_report` to invoke it lazily. **Refuted** because: (a) it leaks Option A's tight coupling back in via a callable indirection; (b) it breaks the "frozen dataclass with declarative state" pattern used by `ReportConfig`; (c) it adds a callable to the determinism stamp surface (closure capture + serialization concerns).

### ¬ß7.3 Validation in `__post_init__` ‚Äî DEFERRED

Adding `__post_init__` validation that warns when `(ic_in_sample, ic_spearman_ci95) == (0.0, (0.0, 0.0))` would surface "wiring gap" signature at construction time. **Defer to F2-T1 Dex discretion** ‚Äî useful diagnostic, NOT load-bearing for verdict correctness (Mira ¬ß4.3 already establishes the empirical signature recognition pattern). If implemented, MUST be a `warnings.warn` at runtime, NOT a `ValueError` (test fixtures currently rely on the all-zero defaults).

## ¬ß8 BacktestResult shape ‚Äî sufficient for IC?

### ¬ß8.1 What's available today

From `packages/vespera_cpcv/result.py`:
- `BacktestResult.fold` ‚Äî `FoldMetadata` with `path_index`, `train_group_ids`, `test_group_ids`, `train_session_dates`, `test_session_dates`.
- `BacktestResult.trades` ‚Äî `tuple[TradeRecord, ...]` with per-trade: `session_date`, `trial_id`, `entry_window_brt`, `direction` (Direction.LONG/SHORT/FLAT), `entry_fill`, `exit_fill`, `pnl_rs`, `slippage_rs_signed`, `fees_rs`, `duration_seconds`, `forced_exit`, `flags`.
- `BacktestResult.metrics` ‚Äî `AggregateMetrics` with `n_trades`, `pnl_rs_total`, `sharpe_daily`, `sortino_daily`, `hit_rate`, `profit_factor`, `max_drawdown_*`, etc.
- `BacktestResult.diagnostics` ‚Äî `FoldDiagnostics` with `n_signals_total`, `n_signals_non_flat`, audit pass/fail flags.

### ¬ß8.2 What's NOT available

IC Spearman is `corr(rank(predictor), rank(label))` over a paired sample. The natural pairings are:
- **Per-trade pairing:** predictor = signal-strength score (e.g., `intraday_flow_magnitude` from `feature_computer.Features`); label = forward return (`pnl_rs / n_contracts / WDO_MULTIPLIER` or normalized signed PnL).
- **Per-fold pairing:** predictor = per-fold `train_sharpe`; label = per-fold `test_sharpe` (this is the `sr_distribution` already computed for DSR in `report.py:494`).
- **Per-trial pairing:** predictor = trial mean Sharpe IS; label = trial mean Sharpe OOS.

`TradeRecord` carries `pnl_rs` (the realized label) and `entry_fill / direction / entry_window_brt` (entry-side data) but **does NOT carry the upstream `Features.intraday_flow_magnitude` predictor score** ‚Äî the score is consumed by `signal_rule.compute_signal` and discarded after the FLAT/LONG/SHORT decision (per `signal_rule.py:1-100`). The information is generated but not persisted into `BacktestResult`.

### ¬ß8.3 Aria recommendation ‚Äî **MINIMAL extension to BacktestResult, NOT mutation**

Add a **new optional field** `BacktestResult.predictor_signal_per_trade: tuple[float, ...] | None = None` (default None for back-compat with all existing tests + persisted artifacts). The field carries the per-trade `intraday_flow_magnitude` (or whatever predictor Mira F2-T0a spec amendment defines) for trades whose `direction != FLAT`. Length matches `len(trades)`.

**Why this is the correct extension:**
- **Back-compat preserved.** Default `None` keeps existing 18+ tests + persisted JSON green; the IC computation handles `None` by falling back to per-fold predictor-pairing using `metrics.sharpe_daily` (which IS available).
- **Article IV trace clean.** Predictor lives next to label (`pnl_rs`) in the same dataclass; provenance is one structural step, not a cross-module reach.
- **Closure body owns the population.** `make_backtest_fn`'s closure body already constructs `Features` per event (`cpcv_harness.py` strategy logic body); persisting `predictor_signal` is a one-line extension to the closure's `TradeRecord` build site. NO factory touch, NO engine touch.
- **Determinism stamp untouched.** The new field is a `tuple[float, ...]` with deterministic content; `BacktestResult.content_sha256()` automatically picks it up via the `_normalize` walk in `result.py:203-219`.

**Alternative (if Mira F2-T0a spec selects per-fold predictor pairing instead of per-trade):** the `compute_ic_from_cpcv_results` function reads `pbo_matrix[trial, fold] vs pbo_matrix_holdout[trial, fold]` (i.e., reuse the per-fold Sharpe IS/OOS pairs already computed inside `compute_full_report` Step 2). In that case, NO `BacktestResult` extension is needed ‚Äî the IC function reads `cpcv_results[trial][i].metrics.sharpe_daily` directly per fold. This is the **lower-risk** path and is the recommended F2-T0a Mira default unless Mira explicitly selects per-trade IC.

### ¬ß8.4 Authority delegation

Mira F2-T0a spec amendment OWNS the predictor‚Üîlabel pairing definition. Aria condition C-A4 below mandates the spec amendment surface explicitly which option (per-trade / per-fold / per-trial) and which predictor extractor to use; the engineering placement (Option D) is invariant to that choice.

## ¬ß9 Anti-leak preservation

The IC pipeline MUST NOT introduce a leakage path that crosses the train‚Üîtest partition boundary established by CPCV.

### ¬ß9.1 D-1 invariant carry-forward

Mira spec ¬ß3.4 mandates `as_of_date == min(test_events.session)` for the per-fold P126 builder; this is the canonical D-1 anti-leak invariant. IC computation under Option D respects this by construction:

- IC predictor is computed per-event INSIDE the closure body (where `Features.intraday_flow_magnitude` already obeys D-1 by construction ‚Äî feature_computer reads only from session snapshot at `entry_ts`, never from future bars).
- IC label is `pnl_rs` from the SAME `TradeRecord` (`result.py:35-52`), generated by the closure's barrier-walk over `[t_decision, vertical_barrier]` ‚Äî strictly forward-only.
- IC aggregation in `compute_ic_from_cpcv_results` consumes `cpcv_results[trial][i].fold.train_session_dates` vs `.test_session_dates` to keep predictor/label pairings WITHIN their original fold partition. No cross-fold mixing.

### ¬ß9.2 IS‚ÜîOOS hold-out lock UNTOUCHED

The hold-out window `[2025-07-01, 2026-04-21]` per `_holdout_lock.assert_holdout_safe` is enforced at `scripts/run_cpcv_dry_run.py:1142-1152` BEFORE any iteration AND inside `CPCVEngine.generate_splits` (T002.0c AC12). Option D adds NO new data-source consumption: it reads `cpcv_results` (post-fanout, in-sample only) ‚Äî the hold-out gate has already fired. Holdout never read in Phase F2.

**ic_in_sample vs ic_holdout decay invariant:** the `evaluate_kill_criteria` K3 decay test (`report.py:594-598`) requires `ic_holdout` from a separate hold-out evaluation. Phase F2 dispatch is **in-sample only** (per Mira sign-off ¬ß6.2 step 2 + ¬ß6.2 footer R6). `ic_holdout` MUST remain `0.0` default in F2 dispatch (or set equal to `ic_in_sample` to make K3 trivially PASS for the in-sample-only run); Mira F2-T0a spec amendment OWNS this choice. Aria condition C-A5 below binds this.

### ¬ß9.3 IC=in_sample alone CANNOT falsify by itself

Mira Gate 4b spec ¬ß1 gates K3 on `IC > 0 AND CI95_lower > 0` ‚Äî strictly in-sample positivity, NOT decay. The IS‚ÜîOOS decay test is K3-secondary (per `evaluate_kill_criteria` defaults) and decoupled from F2 dispatch. F2-T5 re-clearance evaluates K3 as the spec ¬ß1 in-sample positivity test only.

## ¬ß10 Determinism preservation

### ¬ß10.1 Bootstrap CI seed plumbing

`info_coef.bootstrap_ci(sample, statistic, n_resamples, confidence, seed)` already accepts a `seed: int = 42` parameter and uses `np.random.default_rng(np.random.PCG64(seed))` for cross-platform reproducibility (`info_coef.py:100`). Option D forwards `cfg.seed_bootstrap` (default `42`) to both:
- `compute_ic_from_cpcv_results(..., seed_bootstrap=cfg.seed_bootstrap)` ‚Äî bootstrap CI inside IC.
- (future) any DSR bootstrap path inside `compute_full_report` if added.

Single source of truth: `ReportConfig.seed_bootstrap` defined at `report.py:182`. The dry-run script reads `args.seed` (default `42`), passes to `ReportConfig`, which propagates to both consumers.

### ¬ß10.2 Cross-platform reproducibility

PCG64 is platform-stable per numpy ‚â•1.17 (verified by existing `bootstrap_ci` test surface). No new platform-divergent code paths introduced.

### ¬ß10.3 DeterminismStamp impact

`BacktestRunner.build_determinism_stamp` (`runner.py:78-109`) records `seed`, `dataset_sha256`, `spec_sha256`, `cpcv_config_sha256`, `python_version`, `numpy_version`, `pandas_version`, `run_id`, `timestamp_brt`. The IC computation's seed is the same `seed` field already stamped (CPCV seed === bootstrap seed by Mira F2 spec convention; if Mira F2-T0a chooses to decouple, Aria condition C-A6 binds the stamp extension). NO new determinism field required for the default case.

### ¬ß10.4 IC value byte-stability

For any deterministic input `cpcv_results` (which is byte-stable per `BacktestResult.content_sha256()` AC12), the IC computation is a pure reduction with PCG64-seeded bootstrap ‚Üí output IC values are byte-identical across re-runs. This satisfies the AC12 invariant for the new aggregator function.
