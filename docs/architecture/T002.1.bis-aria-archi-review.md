## Decision: APPROVE_OPTION_B

**Canonical:** Option B — `backtest_fn_factory: Callable[[CPCVSplit, pd.DataFrame], Callable]` injected into `CPCVEngine.run`.

**Authority statement:** As `@architect` and DEFERRED-T11 finding M1 owner (2026-04-26 architecture review), I ratify Mira's preference. The factory pattern is the **only** option that simultaneously (1) preserves the closure-purity invariant established by `test_make_backtest_fn::test_closure_independence_no_globals`, (2) places per-fold lifecycle ownership inside the engine where CPCV semantics already live, and (3) keeps T002.0f's six existing AC2 tests green without modification. Options A and C are REFUTED below with structural arch arguments, not preference.

## Reasoning

### Architectural lens — what the engine OWNS vs what the closure OWNS

The current architecture (T002.0f) draws a clean line:

| Owner | Responsibilities | Source |
|-------|------------------|--------|
| **`CPCVEngine`** | Partition, purge, embargo, hold-out lock, per-fold split iteration, result ordering | `packages/vespera_cpcv/engine.py:125-189` |
| **`BacktestRunner`** | DeterminismStamp construction + uniform injection across folds | `packages/vespera_cpcv/runner.py:111-134` |
| **`make_backtest_fn` closure** | Per-event signal→trade→PnL transformation; consumes immutable `(costs, calendar, percentiles_state)` | `packages/t002_eod_unwind/cpcv_harness.py:270-450` |
| **Caller** (`run_5_trial_fanout`, `run_cpcv_dry_run.py`) | Trial-id partitioning + runner construction | `packages/t002_eod_unwind/cpcv_harness.py:495-569`, `scripts/run_cpcv_dry_run.py:707-726` |

DEFERRED-T11 M1 introduces a new responsibility: **rebuilding `Percentiles126dState` between folds**. The architectural question is **which of those four owners** absorbs it.

### Per-option structural analysis

#### Option A — `percentiles_provider` injected into the closure

**What it does:** moves the rebuild **inside** `make_backtest_fn`'s returned closure. The closure body, on every invocation, would call `percentiles_provider(train_events) → Percentiles126dState`.

**Architectural objections:**

1. **Closure purity invariant broken.** `cpcv_harness.py:309-318` documents the contract verbatim: *"The returned closure is pure: no wall-clock, no uuid, no global mutation. Two calls with identical (train, test, split) inputs produce identical `BacktestResult.content_sha256()`"*. Computing percentiles inside the closure means the closure now performs **non-trivial reduction** over `train_events` — formally still pure, but the closure body becomes a 100+ LoC P126 builder instead of a 50 LoC consumer. **Single-responsibility regression.**

2. **Existing test surface breaks.** `tests/cpcv_harness/test_make_backtest_fn.py:46-55, 100, 127, 144-145, 169-170, 186, 197, 217-219, 232-235` all instantiate `make_backtest_fn(BacktestCosts(), _make_calendar(), _make_p126())` — passing a **state**, not a **provider**. Switching to provider semantics breaks the `test_per_fold_p126_rebuild_changes_hash` test by construction (the test currently calls `make_backtest_fn` twice with different states; under Option A it would have to call with different providers, reshaping the entire test file).

3. **Determinism harder to reason about.** Under Option B the closure receives a state with a known `as_of_date` field — the determinism stamp anchors `as_of_date` per fold cleanly. Under Option A, the closure's own invocation triggers the build, so the `Percentiles126dState.computed_at_brt` field becomes a closure-internal wall-clock timestamp. This contaminates `BacktestResult.content_sha256()` unless the provider is **also** time-stamp-injected, which pushes the architectural ugliness to the caller anyway.

4. **Wall-clock cost amplification.** The closure is invoked **once per fold** (45 calls × 5 trials = 225 invocations). Under Option B the rebuild happens **once per fold** (45 × 5 = 225 builds — same cardinality). Under Option A it ALSO happens once per fold, BUT the build is now wrapped inside the closure body — every test that exercises the closure (15+ existing tests) pays the rebuild cost. **No wall-clock advantage** versus Option B; **clear test-time disadvantage**.

**Verdict:** REFUTED.

#### Option B — `backtest_fn_factory` injected into `CPCVEngine.run` (CHOSEN)

**What it does:** the engine, immediately after it produces a `CPCVSplit` (line 184 of `engine.py`), calls `backtest_fn = backtest_fn_factory(split, split.train_events)` to obtain a fold-specific closure, then invokes that closure on `(split.train_events, split.test_events, split)`.

**Architectural arguments FOR:**

1. **Single Responsibility intact.** The engine already owns "iterate over folds and produce per-fold artifacts (purge, embargo, test_blocks)". Adding "produce per-fold closure" is a natural extension of the same responsibility. The closure stays consumer-only: it never builds its own state.

2. **Zero change to `make_backtest_fn` signature** ⇒ all six T002.0f AC2 tests stay green untouched. This is a **load-bearing** property: those tests are the canonical anti-leak proofs (`test_per_fold_p126_rebuild_changes_hash`, `test_closure_independence_no_globals`) — preserving them under refactor preserves the audit trail. **Regression risk near zero.**

3. **CPCV correctness guaranteed by construction.** With the engine owning the factory invocation, there is **no path** by which a fold can accidentally consume another fold's `Percentiles126dState` — the engine produces, hands to factory, factory produces closure, engine invokes closure, result emitted. No shared mutable state. The "global P126 leak" from `cpcv_harness.py:289-292` becomes structurally impossible.

4. **AFML literature alignment (Article IV trace).** Lopez de Prado AFML 2018 §7.4 describes purge+embargo as **per-fold operations**; §12.4 (CPCV) describes the path-aggregation as the engine's responsibility. Beneath that, §12 implicitly assumes **per-fold model retraining** — every CPCV implementation in the wild (mlfinlab, AFML's own examples) treats training-side state as fold-local. The factory pattern is the **canonical** Python idiom for "produce a fresh worker per task" in this regime.

5. **Future Phase F extensibility.** §9 HOLD #2 Gate 5 (Riven dual-sign disarm + capital ramp) will eventually need additional per-fold state (e.g., position sizing calibrated against fold-train Sharpe, regime classifier trained per fold). With Option B, those are **additive parameters to the factory** — the factory signature `Callable[[CPCVSplit, pd.DataFrame], Callable]` is already the right extension point. With Option A, every future per-fold state means another `*_provider` parameter to `make_backtest_fn` ⇒ signature creep.

6. **Parallelization preserved.** The current `CPCVEngine` is documented as thread-safe (`engine.py:106-108`, AC7). The factory invocation per fold is independent across folds; a future `concurrent.futures` driver inside the engine would parallelize trivially. Option C **destroys** this property by moving the loop to the caller.

**Architectural cost (honest accounting):**

- `CPCVEngine.run` signature changes: **one new parameter** (`backtest_fn_factory`), back-compat preserved by making `backtest_fn` and `backtest_fn_factory` mutually exclusive (one OR the other; one required). See §"Implementation guidance for Dex" below.
- `BacktestRunner.run` mirror change: same parameter pair.
- `run_5_trial_fanout` callers (one production caller: `run_cpcv_dry_run.py:711-726`; two test callers in `test_5_trial_fanout.py`) update from `make_backtest_fn(costs, cal, p126)` to a `factory` lambda that calls `Percentiles126dBuilder.build` then `make_backtest_fn`.
- Estimated touched LoC: ~80-120 across 5 files (engine 15-20, runner 15-20, harness 10-15, dry-run script 25-35, test file 10-20). **Bounded, isolated, additive.**

**Verdict:** APPROVED.

#### Option C — caller-managed state, closure stateless

**What it does:** `make_backtest_fn` and `CPCVEngine.run` keep their signatures **untouched**. A new orchestrator (e.g., `run_5_trial_fanout_per_fold` in `cpcv_harness.py`) loops over splits manually, builds per-fold P126 inline, instantiates a fresh closure, calls `CPCVEngine.run(events_filtered, backtest_fn=fold_closure)` **once per fold per trial**.

**Architectural objections:**

1. **CPCV semantics leak out of the engine.** `CPCVEngine.run` becomes a misleading name — it now executes a single fold, not the full CPCV. The 45-path partition + iteration logic gets duplicated in the caller. **Encapsulation violation.**

2. **DeterminismStamp uniformity broken.** `BacktestRunner.run` (`runner.py:126-134`) constructs **one** `DeterminismStamp` per call, then injects it uniformly across all folds via `dataclasses.replace`. Under Option C, the caller calls `runner.run` 45 times per trial ⇒ 45 stamps with **different `run_id`** and **different `timestamp_brt`**. The Beckett T0 handshake (`runner.py:130-134` comment + `run_5_trial_fanout` docstring `cpcv_harness.py:513-518`) explicitly forbids this: *"USE BacktestRunner.run, NOT CPCVEngine.run direct, to preserve uniform DeterminismStamp"*.

3. **Parallelization regression.** Per-fold parallel execution lives **inside** `CPCVEngine` today (and is the natural evolution path). Moving the per-fold loop to the caller forces parallelization to the caller too, breaking the engine's stateless-after-construction contract (`engine.py:106-108`).

4. **CPCV path-id integrity at risk.** The current engine emits splits with `path_id ∈ [0, 44]` in canonical `itertools.combinations` order (`engine.py:155-165`). Under Option C, calling `runner.run` 45 separate times means each call sees a **single path** and the path_id ordering depends on the caller faithfully replicating combinations order. **Risk surface.**

5. **Test surface explosion.** `test_engine_against_afml_benchmark.py` and `test_engine_determinism.py` both rely on `CPCVEngine.run` being **the** CPCV entry point. Option C would force those tests to either (a) test only single folds, losing the C(N,k) coverage check, or (b) wrap a 45-iteration helper, duplicating the engine's responsibility in test code.

**Verdict:** REFUTED.

## Implementation guidance for Dex (binding for T1)

### §I.1 — `CPCVEngine.run` signature update

`packages/vespera_cpcv/engine.py:167-189` — replace the `run` method body with:

```python
def run(
    self,
    events: pd.DataFrame,
    backtest_fn: Callable[[pd.DataFrame, pd.DataFrame, "CPCVSplit"], BacktestResult] | None = None,
    *,
    backtest_fn_factory: Callable[["CPCVSplit", pd.DataFrame], Callable[[pd.DataFrame, pd.DataFrame, "CPCVSplit"], BacktestResult]] | None = None,
    spec_snapshot: dict | None = None,  # noqa: ARG002 — reserved for Beckett §3.1
) -> list[BacktestResult]:
    """Consume ``generate_splits``, invoke per-fold backtest closure.

    Provide EXACTLY ONE of:
      - ``backtest_fn``: a single closure used for all folds (legacy /
        T002.0f stub path; safe when no per-fold state rebuild is needed).
      - ``backtest_fn_factory``: a factory that produces a fresh closure
        per fold from ``(split, split.train_events)``. Required when
        per-fold ``Percentiles126dState`` rebuild is needed (T002.1.bis
        DEFERRED-T11 M1 fix per Aria T0b architectural review).
    """
    if (backtest_fn is None) == (backtest_fn_factory is None):
        raise ValueError(
            "CPCVEngine.run requires EXACTLY ONE of backtest_fn or "
            "backtest_fn_factory to be provided (got "
            f"{'both' if backtest_fn is not None else 'neither'})."
        )
    results: list[BacktestResult] = []
    for split in self.generate_splits(events):
        fn = (
            backtest_fn
            if backtest_fn is not None
            else backtest_fn_factory(split, split.train_events)
        )
        result = fn(split.train_events, split.test_events, split)
        results.append(result)
    results.sort(key=lambda r: r.fold.path_index)
    return results
```

**Back-compat invariant:** all existing `engine.run(events, backtest_fn=...)` callers (production: `runner.py:129`; tests: `test_engine_determinism.py`, `test_engine_against_afml_benchmark.py`, `test_holdout_guard.py`) continue working unchanged. The new factory path is opt-in. AC7 (thread-safety) is preserved — the engine still mutates no shared state; `backtest_fn_factory` MUST itself be thread-safe (caller responsibility — document in docstring).

### §I.2 — `BacktestRunner.run` mirror update

`packages/vespera_cpcv/runner.py:111-134` — add the same `backtest_fn_factory` parameter and forward it verbatim to `self._engine.run`. The `dataclasses.replace` stamp injection (line 133) is **unchanged** — DeterminismStamp uniformity is preserved by construction (one stamp per `runner.run` call, not per fold).

### §I.3 — `run_5_trial_fanout` per-fold rebuild integration

`packages/t002_eod_unwind/cpcv_harness.py:495-569` — refactor `run_5_trial_fanout` to accept either a closure (legacy stub path) OR a factory recipe. Recommended new signature (additive):

def run_5_trial_fanout(
    events: pd.DataFrame,
    runner: BacktestRunner,

The body forwards `backtest_fn` OR `backtest_fn_factory` to `runner.run` per trial.

**Article IV note:** the existing call site `scripts/run_cpcv_dry_run.py:707-726` currently uses positional `(events, backtest_fn, runner)`. Dex MUST migrate that call site **and** the existing tests in `test_5_trial_fanout.py` to keyword args **before** introducing the factory — otherwise the deprecation path is messy. This is a small, isolated rewrite.

### §I.4 — Factory implementation site (consumer)

`scripts/run_cpcv_dry_run.py:707-726` — replace lines 711, 726:

```python
# Per Aria T0b — per-fold P126 rebuild via backtest_fn_factory (Option B).
# DEFERRED-T11 M1 fix: each fold sees its own Percentiles126dState built
# from the train slice, with as_of_date = first_test_session_of_fold.
from packages.t002_eod_unwind.warmup.percentiles_126d_builder import (
    Percentiles126dBuilder,
    DailyMetrics,
)

def _backtest_fn_factory(split: CPCVSplit, train_events: pd.DataFrame):
    # 1. Reduce train_events to per-day DailyMetrics (engineering helper TBD).
    metrics: list[DailyMetrics] = _build_daily_metrics_from_events(train_events)
    # 2. as_of_date = first session of the fold's TEST slice (anti-leak D-1
    #    invariant per spec §3.4 + cpcv_harness.py:309-313).
    as_of_date = min(split.test_events["session"])
    # 3. Build fold-specific Percentiles126dState.
    percentiles_state = Percentiles126dBuilder(calendar).build(
        metrics=metrics,
        as_of_date=as_of_date,
        now_brt=datetime.now(),  # closure stamps via DeterminismStamp anyway
    )
    # 4. Wrap the existing pure closure with per-fold state.
    return make_backtest_fn(costs, calendar, percentiles_state)

cpcv_results = run_5_trial_fanout(
    events,
    runner,
    backtest_fn_factory=_backtest_fn_factory,
)
```

The `_build_daily_metrics_from_events` helper is the **only** new function with non-trivial logic. Its construction is a Mira-binding follow-up: §3.1 of the spec specifies the window construction (`{ DailyMetrics(d) for d in calendar.valid_sample_days if d in train_fold AND d < as_of_date }[-126:]`); Dex implements it as a pure function consuming `train_events` and the `calendar`. This helper lives at `packages/t002_eod_unwind/cpcv_harness.py` (NEW module-level private function `_build_daily_metrics_from_train_events`, ~30-50 LoC, fully unit-testable in isolation).
