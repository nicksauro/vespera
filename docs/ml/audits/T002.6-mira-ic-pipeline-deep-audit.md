### §1.3 Mira semantic adjudication — interpretation **(a)** is the spec-binding one

**Authoritative finding:** the parent thesis H1 alternative + Métrica primária state explicitly:

> `IC_spearman(−sign(ret_acumulado_dia), ret_forward_to_17:55)` — i.e., **a per-event rank correlation between (i) the trial's directional predictor at entry-decision time** (the negative-sign of cumulative-day return as the inventory-unwind fade signal) **and (ii) the forward return realized by 17:55 BRT close** (the strategy's labeled outcome).

This is **interpretation (a)** specialized: per-event, predictor-side = entry-time predictor signal, label-side = forward realized return (or, equivalently after sign-correction, per-event PnL).

**Interpretation (b) is REJECTED** as the primary K3 measurement. Per-trial Sharpe ranking under null hypothesis is **not** what the thesis K3 row encodes; it would be a meta-statistical statement about cross-trial Bonferroni residuals, not the H1 alternative itself.

**Interpretation (c) is REJECTED** as the primary K3 measurement. IS-vs-OOS Sharpe correlation is a PBO-adjacent statistic, not an IC. PBO already covers that ground (BBLP 2014 §3 disjoint folds); the K3 K3-row in the thesis explicitly distinguishes K2 (PBO) from K3 (IC decay).

**Interpretation (c) IS, however, the spec-binding definition of "IC decay"** as a SECONDARY K3 sub-clause: the thesis row K3 explicitly compares `IC hold-out (2025-07→2026-04) < 50% do IC in-sample`. That sub-clause is the IS↔OOS comparison — but it operates on TWO IC values, both of which are computed under interpretation (a). Decay is a derived ratio, not the primary IC measurement.

### §1.4 Final IC semantic specification (Mira authority)

```

    |intraday_flow_magnitude_at_entry| > P_lower_or_upper_for_trial AND

IC_holdout    = IC_spearman(window = hold_out_period   [2025-07-01..2026-04-21])
IC_decay_ratio = IC_holdout / IC_in_sample        (sign-aware; see §4.5)
```

- `IC_in_sample > 0`
- `CI95_lower(IC_in_sample) > 0`  (bootstrap percentile per Efron 1979)
- `IC_decay_ratio >= 0.5`         (thesis K3 row; equivalent to "decay below 50% kills")
- ALSO: rolling-window stability — `decay_per_rolling_window < 1 σ_rolling` (Mira Gate 4b §1 third clause: "decay < 1 σ rolling stdev preserves in-sample sign")

### §2.2 What CPCV emits today (5 trials × 45 paths = 225 results)

- **Per-path Sharpe distribution** (225 values, σ=0.185 on N7 — confirms strategy is alive).
- **Per-trial mean Sharpe distribution** (5 values, feeds DSR `sr_distribution` per Bailey-LdP 2014 eq. 10).

### §2.3 What CPCV does NOT emit today (the wiring gap)

- **Per-event predictor signal at entry-decision time.** The predictor `−intraday_flow_direction_at_entry` is computed inside the strategy logic but is NOT persisted in `TradeRecord`. The closure in `make_backtest_fn` knows the predictor at decision time (it is what triggers entry), but only the resulting `entry_fill` is recorded — not the underlying signal magnitude/sign.
- **Per-event forward-return label.** `ret_forward_to_17:55` is implicit in `pnl_rs` (which is `(exit_price − entry_price) × sign × WDO_MULTIPLIER − costs`), but the raw forward return at the LABEL window (17:55 close, regardless of which barrier actually exited the trade) is NOT persisted. For triple-barrier exits the labeled forward return at 17:55 is what the K3 thesis defines, NOT the realized PnL at exit barrier.

**Critical distinction:** triple-barrier exit price ≠ 17:55 forward-return label. K3 thesis says `ret_forward_to_17:55` — this is the RAW forward return regardless of exit barrier. If a trade exits early at PT/SL, its `pnl_rs` reflects the barrier exit, not the 17:55 close. For pure K3-thesis IC, we need the 17:55 forward return as the label (counterfactual: what would have happened if vertical-time barrier always won).

### §2.5 Required harness extensions (T002.6 F2-T1 scope)

To compute C1 + C2, Phase F2 implementation MUST extend the harness as follows:

1. **`TradeRecord` extension (additive fields, frozen dataclass — semver minor bump):**
   - `predictor_signal: float` — value of `−intraday_flow_direction` at entry-decision timestamp.
   - `predictor_magnitude: float` — value of `|intraday_flow_magnitude|` at entry timestamp.
   - `forward_return_at_1755_pts: float` — close-to-close forward return from `entry_fill.ts` to `17:55:00 BRT` of `entry_fill.ts.date()`, in WDO points. This is the C1 label and is computed at trade-record-time by reading the session tape close-price at the 17:55 boundary.

2. **Strategy closure extension (`make_backtest_fn`):**
   - At entry-decision time, capture `predictor_signal`, `predictor_magnitude` directly from the signal evaluation (already computed for the signal rule; just persist).
   - At trade-record-time, lookup session-tape's price at `17:55:00 BRT` of `entry_session_date` (or last trade ≤ 17:55:00 BRT per Nova spec §6 yaml block `auction.exit_price_rule`); compute forward return from `entry_fill.price` to that close; persist as `forward_return_at_1755_pts`.

3. **`AggregateMetrics` is NOT extended.** IC is computed over the *flat-by-trade* event vector pooled across all paths in a window; per-fold metrics is the wrong granularity for IC.

These three additions are **additive** (do not break existing fields) — frozen dataclass minor-version compat. The factory signature, per-fold P126 D-1 invariant, and CPCV split semantics are UNCHANGED.

### §2.6 Where IC is computed in the pipeline

Given §2.5 additions, the IC computation lives at the **report-aggregation layer**, NOT inside the per-fold strategy closure:

Phase F2 fix is to insert a small computation site there. Concrete spec in §4 below.

### §2.7 Dedup invariant — one event one count

Each (session_date, trial_id, entry_window_brt) tuple is one event. Across CPCV's 45 paths × 5 trials, each event is counted in **one and only one test fold per trial** by CPCV construction (each session belongs to exactly one of the N_groups disjoint partitions, and `test_group_ids ⊂ groups` means the session is in exactly one test set per path). However, the same session appears in multiple `path_index` test sets across the C(N,k) combinatorial paths.

**Dedup rule (Mira §2.7 binding):** when assembling the `(predictor_vector, label_vector)` from `cpcv_results` for IC computation, group by `(session_date, trial_id, entry_window_brt)`; take the FIRST occurrence (sorted by `path_index` ascending). This avoids inflating N by counting the same event multiple times across paths.

Expected `N_unique_events` for in-sample 2024-08-22..2025-06-30: ~225 sessions × 5 trials × 4 entry windows × signal-fired-rate ≈ 3000-3800 events (matches Beckett N7 full report's `n_events=3800`).

**N >= 30 floor per Bailey-LdP 2014 §3** trivially satisfied at this scale (3800 ≫ 30). Bootstrap CI is well-conditioned.

### §3.3 Bootstrap-of-IC — wiring contract for paired resamples

The current `bootstrap_ci` takes a 1-D sample + a statistic-of-1-D-sample. For IC bootstrap we need to bootstrap on PAIRED resamples — bootstrapping the predictor and label INDEPENDENTLY destroys the correlation. Phase F2 wiring contract:

```python
def ic_bootstrap_ci(
    predictor: np.ndarray,
    label: np.ndarray,
    n_resamples: int = 10_000,
    confidence: float = 0.95,
    seed: int = 42,
) -> tuple[float, float]:
    """Paired-bootstrap percentile CI for ic_spearman(predictor, label).

    For each resample, draw N indices WITH REPLACEMENT and resample BOTH
    predictor and label using the SAME indices (paired). Compute IC on
    the paired resample. Percentile CI over n_resamples IC values.

    Determinism via np.random.PCG64(seed). Cross-platform stable.
    Edge case: if predictor or label has zero variance, returns (0.0, 0.0)
    matching ic_spearman semantics for degenerate inputs.
    """
    n = predictor.size
    if n != label.size:
        raise ValueError("predictor and label must have same length")
    if n < 2:
        raise ValueError("ic_bootstrap_ci requires N >= 2")
    rng = np.random.default_rng(np.random.PCG64(seed))
    idx = rng.integers(0, n, size=(n_resamples, n))
    ic_samples = np.empty(n_resamples, dtype=float)
    for r in range(n_resamples):
        i = idx[r]
        ic_samples[r] = ic_spearman(predictor[i], label[i])
    alpha = 1.0 - confidence
    lo = float(np.percentile(ic_samples, 100.0 * alpha / 2.0))
    hi = float(np.percentile(ic_samples, 100.0 * (1.0 - alpha / 2.0)))
    return lo, hi
```

This is a NEW helper to be added to `packages/vespera_metrics/info_coef.py` in Phase F2 implementation. It REUSES `ic_spearman()` and is therefore self-consistent with the existing T002.0d spec §3 contract. Determinism witness via paired PCG64 indices; cross-platform stability inherited from `ic_spearman` + `np.random.PCG64`.

`n_resamples = 10_000` is the T002.0d Mira spec §3 default; PRESERVED. `seed` in Phase F2 wiring threads through `ReportConfig.seed_bootstrap` (already exists; T002.0d AC11 verified).

### §4.5 Decay metric specification (deferred to Phase G hold-out unlock)

K3 decay is a derived ratio:

```

```

Under Phase F2 (hold-out lock active), `IC_holdout` is NOT computed; therefore decay is NOT computed; therefore the K3 PASS condition for Phase F2 narrows to the in-sample-only triplet `IC_in_sample > 0 AND CI95_lower > 0` per Mira Gate 4b spec §1 K3 row strict reading (§1.4 above). The decay sub-clause (`< 50% in-sample`) and the rolling-window stability sub-clause (`< 1 σ rolling`) are deferred to Phase G as part of hold-out unlock procedure (separate Riven authority + separate spec).

`IC_holdout` = 0.0 in Phase F2 reports is a SENTINEL value that — combined with the NEW provenance field `MetricsResult.ic_holdout_status="holdout_locked"` (§4.7) — explicitly distinguishes "deferred (locked)" from "computed and is zero". Article IV: NEVER conflate "not computed under hold-out lock" with "computed and is zero" — the failure mode of Round 1 was exactly this conflation at the default-value level.

### §4.6 Sign-aware decay (Phase G deferred — recorded for completeness)

When Phase G unlock fires and `IC_holdout` is computed:
- If `sign(IC_in_sample) ≠ sign(IC_holdout)`: decay ratio is meaningless (sign flip = effect inverted, not decayed). Report as `IC_decay_ratio = NaN; sign_flipped=True; verdict=K3_FAIL_sign_flip`.
- If `sign(IC_in_sample) == sign(IC_holdout)`:
  - If `|IC_holdout| / |IC_in_sample| < 0.5`: K3 FAIL decay.
  - Else: K3 PASS decay.

This sub-spec is a Phase G future-work artifact; surfaced here only so Phase F2 spec amendment §6 below is complete-by-design.

## §5 Audit gap-acceptance failure — self-audit

This section is the hardest part of the audit. As the author of the T002.0d Mira spec + the Mira Gate 4b spec, I have the obligation under Article IV to identify WHY a verdict-issuing pipeline shipped with a default `ic_in_sample = 0.0` that flowed through to a `KillDecision` evaluation without anyone — me, Quinn, Beckett, the Pax Q-SDC validation, the Aria architectural review — flagging it as a verdict-binding gap.

### §5.1 T002.0d Mira spec drafting blind spot

**The blind spot:** `T002.0d` Mira spec §1-§9 specified the IC measurement function (`ic_spearman`, `bootstrap_ci`) and the report-aggregation contract (`MetricsResult`, `ReportConfig`, `evaluate_kill_criteria`) — but it did NOT specify the **computation site** that produces the IC values fed into `ReportConfig`. The spec assumed an upstream caller would supply them, and the docstring of `ReportConfig` even says so verbatim:

```python
ic_in_sample: float = 0.0
# IC values are caller-supplied — no synthetic default that would
# hide K3 behind invented numbers (Article IV).
ic_holdout: float = 0.0
```

The docstring is correct in intent (Article IV: do NOT invent a synthetic default that hides K3). But the design provided a `0.0` default WITHOUT a guard that fail-closes when no caller supplies a value. The `0.0` default is itself a synthetic value — it just happens to coincide with the K3 FAIL threshold (`IC > 0` is failed by `0.0`), which means the report TECHNICALLY emits a K3 FAIL when no caller provides IC data, BUT WITHOUT signaling that the FAIL is due to "not computed" rather than "computed and zero".

**Specifically what the spec should have mandated (and did not):**

1. A `MetricsResult.ic_status` provenance field distinguishing "computed" from "not computed" (§4.7 above — NEW in Phase F2).
2. An invariant on `compute_full_report`: when `ReportConfig.ic_in_sample == 0.0` AND `ic_spearman_ci95 == (0.0, 0.0)`, raise `NotComputed` error UNLESS the caller explicitly opted-in via a `cfg.ic_status="not_computed"` declaration. This makes the silent fall-through into the K3 verdict an explicit caller decision, not a default.
3. A unit test in `test_compute_full_report.py` that asserts: when the caller does NOT supply IC values, `compute_full_report` either raises (preferred) OR emits a `MetricsResult` with `ic_status != "computed"` and `evaluate_kill_criteria` then HONORS that status by emitting a NEW reason `K3_NOT_COMPUTED — ic_status='not_computed'` rather than the misleading `K3: IC_in_sample=0.000000 non-positive — no edge`.

### §5.2 Quinn QA Gate T002.0d blind spot

**The blind spot:** Quinn QA Gate `docs/qa/gates/T002.0d-qa-gate.md` 8 mandatory checks + AC1-AC14 traceability matrix verified that `ic_spearman()`, `bootstrap_ci()`, and `evaluate_kill_criteria()` were each individually correct (AC5, AC6, AC11, AC14 — all PASS). The gate did NOT verify that the upstream **wiring** from `cpcv_results` to `ReportConfig.ic_*` exists. AC12/AC12.1/AC13 (the integration test that would have caught the gap) were EXPLICITLY DEFERRED in the gate (`docs/qa/gates/T002.0d-qa-gate.md` §3 AC12/AC12.1/AC13 row: "DEFERRED — T002.0c handshake gate — Beckett `BacktestResult` interface in flight").

**Specifically what the QA gate should have caught (and did not):**

The deferral note correctly stated the Beckett `BacktestResult` interface was in flight; that fact is real. What the QA gate should have ALSO required was a follow-up integration-test gate added to the T002 critical path, scheduled for closure BEFORE the first verdict-issuing CPCV dry-run. Instead, the deferral became a permanent gap — by the time T002.0f T2 + T002.0g + T002.0h + T002.1.bis closed, the integration test for IC pipeline wiring was never re-opened, and Beckett N7 was permitted to issue a `KillDecision` without it.

The fix is structural, not blame-assigning: the QA-gate template needs an **explicit register-and-close field for deferred ACs that gate verdict authority**, so a deferral pre-N7 cannot ship a verdict-issuing run without first closing the deferred gate.

### §5.3 T002.0f wiring authoring blind spot

**The blind spot:** `compute_full_report` (T002.0f T2) wires the existing IC values from `ReportConfig` to `MetricsResult` and to `evaluate_kill_criteria` correctly. The author (Dex) had no signal that an UPSTREAM caller wiring was missing — the spec said "caller supplies", and the function correctly consumes what the caller supplies. This is not a Dex blind spot; it is a spec-handoff blind spot upstream.

### §5.4 T002.0f T3 dry-run executor authoring blind spot

**The blind spot:** `scripts/run_cpcv_dry_run.py:1070` constructs `ReportConfig(seed_bootstrap=seed)` — i.e., the executor knows about `seed_bootstrap` but does NOT know that it is also responsible for computing IC values. The story T002.0f T3 ACs (per `docs/stories/T002.0f.story.md` AC list) prescribed exit codes, telemetry, hold-out lock, sha256 stamping, and `compute_full_report` invocation — but not "compute IC values upstream of the report".

**Specifically what the story spec should have included (and did not):**

1. An AC explicitly tasking the dry-run executor with computing IC at the report-aggregation layer.
2. A unit test asserting that `_run_phase` populates `cfg.ic_in_sample > 0` (or fails gracefully with a documented inconclusive verdict — NOT silently emit `0.0`).

### §5.5 Test fixture gap

`tests/vespera_metrics/test_compute_full_report.py` has fixtures with `ic_in_sample=0.10, ic_holdout=0.08` (line 652-653) — i.e., the integration tests for `compute_full_report` SET the IC values explicitly. There is NO test fixture that verifies the WIRING (i.e., end-to-end from `cpcv_results` to a non-zero `MetricsResult.ic_spearman`). This is the test-fixture gap; the unit tests covered the unit, but no integration test covered the SEAM between `run_5_trial_fanout` output and `compute_full_report` input WRT the IC fields.

### §5.6 Verdict-issuing protocol blind spot

**Most important meta-finding:** the squad's verdict-issuing protocol does not currently REQUIRE that every numeric metric reported in a `KillDecision` be verifiable as "honestly computed" from the run-time data. The Beckett N7 full report made the right call — Beckett surfaced two interpretations (A: real edge negative; B: implementation artifact) and DEFERRED to Mira T5 authority. But the GATE process itself permitted a `kill_decision.k1_dsr_passed=true / k3_ic_decay_passed=false` to be SERIALIZED to `full_report.json` without a synchronous "is K3 honestly measured?" check.

**The fix is process + spec:** Phase F2 §15.5 below introduces an Anti-Article-IV Guard #8 that requires every verdict-issuing report to carry per-metric `*_status` provenance flags ("computed" / "deferred" / "not_computed"); a verdict layer that consumes a `*_status != "computed"` flag MUST emit `K_NOT_COMPUTED` rather than falsely emitting `K_FAIL`. The Round 1 "GATE_4_FAIL_data_quality" verdict held up under stricter inspection (§4 of the T5 sign-off) because Mira manually traced the K3=0 to a wiring gap; under Phase F2 process, the trace would be automatic via the `ic_status="not_computed"` flag and the FAIL would be re-cast as `INCONCLUSIVE_K3_NOT_COMPUTED` from the start.
