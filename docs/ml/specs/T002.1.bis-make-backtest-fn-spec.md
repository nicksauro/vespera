### В§1.1 вЂ” Per-event invocation contract

For each row `(t_start, session, trial_id, entry_window)` in the `test_events` DataFrame produced by `build_5_trial_events_df` (cpcv_harness.py:267-264), `make_backtest_fn`'s closure body executes **exactly one trade decision**:


**The closure NEVER iterates beyond a single (t_start, trial_id) row.** Cross-event accounting (sample uniqueness, Sharpe per fold) is the engine's responsibility, not the closure's.

## В§2 вЂ” Triple-barrier exit semantics

**Source of truth:** spec yaml L70-74 (`label_alt`).

### В§2.1 вЂ” Three barriers (Lopez de Prado AFML 2018 В§3.4)

Given entry fill `(entry_ts, entry_price)`:

| Barrier | Formula | Trigger condition |
|---------|---------|------------------|
| **Profit target (PT)** вЂ” upper for LONG, lower for SHORT | `pt_offset = 1.5 * ATR_hora` (spec L72) | First trade tick where `(side == LONG and tick_price >= entry_price + pt_offset)` OR `(side == SHORT and tick_price <= entry_price - pt_offset)` |
| **Stop loss (SL)** вЂ” lower for LONG, upper for SHORT | `sl_offset = 1.0 * ATR_hora` (spec L73) | First tick where `(side == LONG and tick_price <= entry_price - sl_offset)` OR `(side == SHORT and tick_price >= entry_price + sl_offset)` |
| **Time horizon (vertical)** | `t_vertical = 17:55:00 BRT same session` (spec L74) | Hard stop if no PT/SL hit before `t_vertical`; mid-price at `t_vertical` is the fill price |

### В§2.2 вЂ” `ATR_hora` definition

Per spec yaml L72-73: `ATR_hora` is **hourly ATR** (not 20d daily ATR). Mira ruling resolves the ambiguity:

```
ATR_hora(session, t) = mean(true_range_15min[s, t-4..t])  # rolling last 4 Г— 15-min bars
```

where `true_range_15min` is computed over 15-minute bars from session open 09:30 BRT. This gives ~4 bars/hour, matching standard market-microstructure ATR practice.

**Computability against historical-availability (Nova trades-only constraint):** PASS вЂ” true range from 15-min OHLC is computable from trade tape alone (no book required).

**Defensive fallback:** if `ATR_hora` is undefined (first hour of session, < 4 bars) в‡’ **DO NOT TRADE**. The signal_rule's existing `regime_atr_undefined` flag covers this; closure checks `Features.flags` and emits FLAT.

### В§2.3 вЂ” Barrier-hit precedence rules

When two barriers fire on the same tick (rare but possible on gaps):

```
Precedence: SL > PT > vertical
```

**Rationale (Mira primary literature anchor, Lopez de Prado AFML 2018 В§3.4):** stop-loss first preserves risk-management invariant; PT second; vertical only fires when neither other barrier triggered. This is the canonical AFML В§3.4 ordering.

### В§2.4 вЂ” Barrier-hit timestamping

`exit_ts = first_tick_ts where barrier triggered` (NOT mid-price ts). This is the **earliest possible** exit; engine attributes this trade to the test fold containing `t_start` (CPCV embargo guarantees `exit_ts < embargo_window` is held by upstream split logic, NOT enforced inside the closure).

**Anti-leak invariant:** the closure NEVER consults trade ticks beyond `t_vertical = 17:55 BRT same session as entry`. `exit_ts в€€ [entry_ts, 17:55]`.

## В§3 вЂ” P60 / P20 / P80 percentile filter activation rules

**Source of truth:** spec yaml L80-82 + cpcv_harness.py:309-313 (anti-leak invariant docstring).

### В§3.1 вЂ” Filter calculation

`PercentileBands` come from `Percentiles126dState` (warmup builder, percentiles_126d_builder.py:91+). For the train fold producing the bands:

```
window = sorted({ DailyMetrics(d) for d in calendar.valid_sample_days
                                  if d in train_fold AND d < as_of_date })[-126:]
bands.magnitude     = (P20, P60, P80) over window
bands.atr_day_ratio = (P20, P60, P80) over window
```

**Critical anti-leak:** `as_of_date` for fold k is the **first session of fold k's test slice**, NOT the global in_sample start. Each fold sees a **distinct** `Percentiles126dState`. This is the **DEFERRED-T11 M1 fix** (В§6).

### В§3.2 вЂ” Activation triggers

Per spec yaml L80-81 + signal_rule.py:75-85:

| Trigger | Condition | Action |
|---------|-----------|--------|
| **Entry** | `Features.intraday_flow_magnitude > bands.magnitude.{p60|p50|p70}` (per trial) | Proceed to regime check |
| **Regime filter** (T1/T2/T3/T5 only вЂ” T4 disables) | `bands.atr_day_ratio.p20 в‰¤ Features.atr_day_ratio в‰¤ bands.atr_day_ratio.p80` | Proceed to fade direction |
| **Direction** | per В§1.3 | Emit Signal |
| **Failure (any)** | else | `Signal(FLAT, <reason>)` |

### В§3.3 вЂ” Tie-break determinism

`Percentiles126dBuilder._percentile` uses numpy type-7 linear interpolation (percentiles_126d_builder.py:139-153). Ties on bands are **deterministic** вЂ” same input window в‡’ same bands в‡’ same signal. Closure preserves this; introduces no random shuffles, no `np.random` calls.

### В§3.4 вЂ” D-1 invariant

Per cpcv_harness.py:309-313 + percentiles_126d_builder.py:131-136 (`m.day < as_of_date` filter): bands are computed over days STRICTLY BEFORE the entry session. Day-of-entry is **not in the percentile window**, eliminating same-day leakage.

### В§5.2 вЂ” IR DARF treatment (Mira ruling)

Per atlas L210-225 (`treatment_choice: post_hoc_on_monthly_net_gain`):

**The closure does NOT compute IR DARF.** Per-trade PnL is **gross** of IR. Tax is applied at the Sharpe-distribution-aggregation layer (out of closure scope), preserving Lopez de Prado В§3 DSR formula (gross-of-tax cohort statistics вЂ” adding tax distorts the per-trade return distribution non-linearly).

**For T002.1.bis Beckett N6 verification (AC11):** Beckett's report N6 В§X documents the gross PnL distribution AND a separate post-hoc table of monthly-aggregated IR-net PnL. Mira clearance Gate 4 metrics (DSR, PBO) are computed on **gross PnL** (research convention, Bailey-LdP 2014 В§3); production capital ramp will use IR-net (Riven Gate 5 prep).

### В§5.3 вЂ” PnL formula (closure-internal)

Per trade `i`:

```
gross_pnl_rs[i] = (exit_price - entry_price) * sign(direction) * n_contracts * WDO_MULTIPLIER
                  - 2 * n_contracts * (brokerage_per_contract_side_rs + exchange_fees_per_contract_side_rs)
                  # slippage already baked into entry_price + exit_price via BacktestBroker._slippage()
```

**WDO_MULTIPLIER = R$ 10.00/ponto** вЂ” single-source from `DOMAIN_GLOSSARY.md#wdo-mini-dГіlar-futuro` per fix G005 (spec yaml L394). Hardcoded constant `WDO_MULTIPLIER = 10.0` already in exec_backtest.py:29 вЂ” closure consumes verbatim.

### В§5.4 вЂ” Cost atlas SHA + version locks

Per `from_engine_config(...)` (exec_backtest.py:114-130): atlas SHA256 (LF-normalized) and `atlas_version` are validated at config-load time. Closure inherits the validated `BacktestCosts`. **NO atlas-side validation inside closure** вЂ” done once at engine init.

## В§6 вЂ” Per-fold P126 rebuild architecture (DEFERRED-T11 M1 fix)

**Source of truth:** cpcv_harness.py:287-307 docstring + Aria 2026-04-26 architecture review finding M1.

This section is **architectural** вЂ” Aria T0b sign-off **chooses among** the three options below. Mira's role here is to enumerate alternatives + state preference; Aria authority decides the canonical.

### В§6.1 вЂ” Mira preferred design: Option B (`backtest_fn_factory`)

**Signature:**

```python
def CPCVEngine.run(
    events: pd.DataFrame,
    splits: Sequence[CPCVSplit],
    backtest_fn_factory: Callable[[CPCVSplit, pd.DataFrame], Callable[[pd.DataFrame, pd.DataFrame, CPCVSplit], BacktestResult]],
    ...
) -> Sequence[BacktestResult]:
    for split in splits:
        train_events = events.iloc[split.train_indices]
        backtest_fn = backtest_fn_factory(split, train_events)   # в†ђ per-fold closure
        results.append(backtest_fn(train_events, test_events, split))
```

**Where the factory is responsible for:**
1. Receiving `train_events` (rows of fold k's train slice).
2. Computing the per-day `DailyMetrics` over `train_events`.
3. Calling `Percentiles126dBuilder.build(metrics, as_of_date=first_test_session, ...)` to get the fold-specific `Percentiles126dState`.
4. Calling `make_backtest_fn(costs, calendar, percentiles_state)` to produce the closure.
5. Returning the closure.

**Why Mira prefers Option B:**
- **Engine becomes the orchestrator** of the per-fold lifecycle (single responsibility).
- **`make_backtest_fn` keeps its current pure-function signature** вЂ” already passes `test_per_fold_p126_rebuild` because re-calling it produces a new closure with a new `Percentiles126dState`.
- **No change to `make_backtest_fn` signature** в‡’ minimal regression risk; existing tests stay green.
- **Factory pattern is canonical** in CPCV literature (Lopez de Prado AFML В§7 and В§12) вЂ” N(splits) closures, each scoped to a single split.

### В§6.2 вЂ” Alternative design: Option A (`percentiles_provider` injected into closure)

**Signature:**

```python
def make_backtest_fn(
    costs: BacktestCosts,
    calendar: CalendarData,
    percentiles_provider: Callable[[pd.DataFrame], Percentiles126dState],
) -> Callable[..., BacktestResult]:
    def backtest_fn(train_events, test_events, split):
        # rebuild bands from THIS fold's train slice
        percentiles_state = percentiles_provider(train_events)
        ...
    return backtest_fn
```

**Tradeoffs vs Option B:**
- Pro: only one `make_backtest_fn` invocation per run; one closure; less factory-pattern overhead.
- **Con: rebuild happens INSIDE the closure body** вЂ” closure is no longer pure (it computes percentiles) and is harder to unit-test (must mock the provider).
- Con: `make_backtest_fn` signature changes (breaks `test_per_fold_p126_rebuild` which currently re-instantiates).
- Con: `percentiles_state` enters and leaves the closure scope per call вЂ” captures train_events implicitly per invocation, harder to reason about determinism.

### В§6.3 вЂ” Alternative design: Option C (state caller-managed, closure stateless)

**Signature:** `make_backtest_fn` keeps current signature **unchanged**; the **caller** (e.g., a new `run_5_trial_fanout_per_fold` orchestrator at cpcv_harness.py:475+) manually loops over splits, rebuilds `Percentiles126dState`, and calls `CPCVEngine.run` once per split with a freshly-instantiated closure.

**Tradeoffs:**
- Pro: zero changes to `CPCVEngine.run` signature; zero changes to `make_backtest_fn` signature.
- Con: `CPCVEngine.run` becomes a per-split helper, not a CPCV engine. Loses bulk parallelization potential.
- Con: caller has to manually orchestrate splits вЂ” duplicated CPCV iteration logic across consumers.
- Con: weakest separation of concerns; engine no longer owns CPCV semantics.

### В§6.4 вЂ” Mira recommendation summary

| Option | Engine signature change | `make_backtest_fn` signature change | CPCV correctness | Mira preference |
|--------|------------------------|--------------------------------------|------------------|----------------|
| **B (factory)** | YES (`backtest_fn_factory` param) | NO | strongest | **PREFERRED** |
| A (provider injected) | NO | YES (breaking) | acceptable | secondary |
| C (caller-managed) | NO | NO | weakest | last resort |

**Aria T0b authority decides.** Mira spec is binding on В§1-В§5, В§7; binding on the existence of a per-fold rebuild mechanism; **non-binding** on which of A/B/C is canonical (Aria chooses).

## В§7 вЂ” Toy benchmark (Bailey-LdP 2014 reproducibility anchor)

**Source of truth:** Bailey & Lopez de Prado, "The Deflated Sharpe Ratio: Correcting for Selection Bias, Backtest Overfitting, and Non-Normality" (Journal of Portfolio Management, Vol. 40, No. 5, 2014), В§3 (DSR formula derivation) + Table 2 (toy example calibration).

### В§7.1 вЂ” Toy strategy A: pure noise (no edge)

**Construction:**
- Synthetic per-trade returns drawn iid from `N(Ој=0, Пѓ=1)`.
- N=45 paths Г— n_trades_per_path=20 (matches T002 CPCV 45-path Г— ~20 events/path order of magnitude).
- n_trials=5 (Bonferroni preserved).

**Bailey-LdP 2014 В§3 expected metrics:**
- **`SR_estimate в‰€ 0.0`** вЂ” noise has no skill; per-path Sharpe centered on zero.
- **`Пѓ(SR_distribution) в‰€ 1/sqrt(n_trades) в‰€ 0.22`** вЂ” sampling spread, not skill.
- **`DSR в‰€ 0.5`** вЂ” by construction; null distribution gives 50% probability skill > 0.
- **`PBO в‰€ 0.5`** вЂ” Bailey-Borwein-LdP 2014: under H0 (no skill), best-IS-Sharpe is uniformly distributed across rank-OOS positions в‡’ PBO = 0.5.

**Acceptance:** `make_backtest_fn` over toy strategy A reproduces these to within В±0.05 (sampling tolerance).

### В§7.2 вЂ” Toy strategy B: small edge

**Construction:**
- Synthetic per-trade returns drawn iid from `N(Ој=0.20, Пѓ=1)` вЂ” small positive edge (Sharpe в‰€ 0.20 per trade).
- N=45 paths Г— n_trades_per_path=20.
- Annualized SR (assuming 252 trading days, 1 trade/day): `SR_annual в‰€ 0.20 * sqrt(252) в‰€ 3.17`.

**Bailey-LdP 2014 В§3 expected metrics:**
- **`SR_distribution_mean в‰€ 0.20`** per-trade (not annualized).
- **`DSR > 0.95`** with n_trials=5 Bonferroni (`p_adj = 0.01/5 = 0.002`) вЂ” skill survives multiple-testing penalty.
- **`PBO < 0.25`** вЂ” skill is robust across IS/OOS rank decomposition.

**Acceptance:** `make_backtest_fn` over toy strategy B reproduces these to within В±0.05.
