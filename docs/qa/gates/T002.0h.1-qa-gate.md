#### LOW-1 — `events_metadata["warmup_gate_as_of"]` indirection

- **Severity:** low
- **Category:** code
- **File:** `scripts/run_cpcv_dry_run.py:926`
- **Description:** The `events_metadata` dict (returned by `_run_phase`) sets `"warmup_gate_as_of": in_sample_start.isoformat()` rather than `as_of_date.isoformat()`. In the current call sites this is functionally equivalent — `main` invokes `_run_phase(in_sample_start=smoke_start, as_of_date=smoke_start, ...)` for smoke and `_run_phase(in_sample_start=in_sample_start, as_of_date=in_sample_start, ...)` for full, so the two values coincide. However, the field name claims it represents the gate's as_of, which is now a separate parameter; future refactors that decouple `as_of_date` from `in_sample_start` (e.g., custom warmup as_of overrides) would silently drift this metadata field.
- **Authority cross-reference:** Mira T0 explicitly noted this in story L398 + L415: "the `events_metadata["warmup_gate_as_of"]` field (L761) becomes phase-correct (currently uses `in_sample_start.isoformat()` which is already per-phase since `_run_phase` receives phase-specific `in_sample_start`)." Mira's guidance #1 was "**MUST** preserve existing `events_metadata["warmup_gate_as_of"] = in_sample_start.isoformat()` — this field is **already phase-correct**. Do not alter." → Dex compliance HONORED.
- **Recommendation:** ACCEPT current behavior per Mira T0 explicit guidance. No action required at this gate. **Recommended for T002.1.bis follow-up:** if/when the gate's `as_of_date` decouples from `in_sample_start` (e.g., for per-fold P126 rebuild semantics), update this field to `as_of_date.isoformat()` or rename to `phase_window_start_brt` to disambiguate.
