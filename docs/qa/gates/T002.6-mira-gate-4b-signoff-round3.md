### §3.3 Lenient reading rejected

User task hint § Issue #1 frames a "lenient reading": "N7-prime já tinha IC_IS=0.866010 measured + N8 já measurou IC sobre holdout window = 0.865933 → compute decay manually = 0.866 > 0.433 → K3 decay PASS empirically." Mira Round 3 considers and **rejects** this as the binding K3 decay test for the following reasons:

1. **Cross-window vs same-window IC semantics.** F2 N7-prime IC_in_sample=0.866010 is the rank-correlation `(predictor=-intraday_flow_direction, label=ret_forward_to_17:55_pts)` over events ∈ in-sample window 2024-08-22..2025-06-30. N8 IC=0.865933 is the rank-correlation over events ∈ holdout window 2025-07-01..2026-04-02. Both are *in-sample-IC-over-their-respective-window* measurements. Neither is the canonical decay metric §15.13.5 demands — that metric (IC_holdout in the spec text) is the rank-correlation **over the holdout window using a model trained on the in-sample window**, where the model's predictor↔label binding is the in-sample-fit binding extrapolated forward. Here the predictor↔label binding is non-parametric (a rank correlation, not a learned model), so the in-sample/out-of-sample distinction collapses — the "IC_holdout" value is just "IC over a different time window with the same definition." This is statistically informative but is **not** the AFML §8.6 IC stability rolling-decay paradigm Mira spec §15.13.5 invokes.

2. **Bootstrap procedure non-equivalence.** F2-T1 IC pipeline bootstrap (`paired-resample bootstrap n=10000 PCG64 seed=42`, spec §15.4) is performed **within** a single window. Cross-window decay is NOT bootstrap-comparable across windows under the current implementation (N8 produced an in-sample-IC-over-holdout-window with its own n=10000 bootstrap CI95 [0.864880, 0.866025], width 0.001145 — non-degenerate, comparable F2 N7-prime CI width 0.000738). To produce a binding K3 decay test, the IC_holdout would need to use the in-sample model's PREDICTIONS evaluated over OOS labels (a deliberately cross-window measurement). The current N8 measurement is not that.

3. **PRR §3.1 R15 is HASH-FROZEN; the trigger conjunction is text-binding.** `IC_holdout > 0.5 × IC_in_sample` was registered with both terms semantically distinct (one over IS window, one over OOS window with proper unlock protocol). Re-interpreting them as same-window-different-times to back out a manual PASS would constitute **post-hoc disposition rule mutation** (Article IV violation; ESC-012 R3 + Anti-Article-IV Guard #4 collective). The PRR was designed precisely to forbid this kind of reinterpretation under outcome pressure.

4. **§15.13.5 explicit text** says "IC_holdout magnitude must preserve in-sample IC sign AND retain ≥ 50% of in-sample magnitude."
