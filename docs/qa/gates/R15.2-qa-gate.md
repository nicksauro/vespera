## Findings

### HIGH — CODEOWNERS uses AIOX agent IDs, not real GitHub handles
- **Location:** `CODEOWNERS` (root) lines 15, 18, 21, 24
- **Issue:** `@dev`, `@risk-manager`, `@devops`, `@data-engineer` are AIOX persona IDs, not GitHub usernames or team mentions. GitHub will silently fail to route review requests to non-existent handles. This nullifies the L3 enforcement layer of defense-in-depth.
- **Impact:** L1 (hook) + L2 (CI) layers remain intact; L3 (CODEOWNERS-required reviews) becomes advisory-only when branch protection is eventually toggled.
- **Mitigation acknowledged:** Story Dev Agent Record §T5 already flags "AIOX agent IDs may need swap to actual GH handles/teams pre-T6." Header note in CODEOWNERS itself (lines 11-12) acknowledges the swap requirement.
- **Recommendation:** Gage @devops must replace AIOX IDs with real GitHub handles (or `@org/team-*` mentions) BEFORE T6 branch-protection toggle. **Not a Quinn blocker** — falls within Gage R12 scope as part of T6 prep.

### MEDIUM — T6 (Gage R12 branch-protection toggle) externally blocked
- **Location:** AC9, mission constraint
- **Issue:** GitHub Free tier does not support branch-protection API on private repos. Without T6, L2 CI is advisory-only on `main` (red checks visible but not merge-blocking).
- **Impact:** Defense-in-depth degrades from L1+L2+L3 to L1+L2-advisory (still meaningful — L1 hook + visible-but-non-blocking L2 CI). Bypass window: developer with `--no-verify` AND ignore-CI-red could land canonical mutation without cosign. Risk profile: ALTA in absence of T6.
- **Recommendation:** Escalate to user — options are (a) upgrade repo to Pro/Team tier; (b) make repo public (free branch protection on public repos); (c) accept advisory-only L2 + rely on L1 hook discipline + post-merge audit via Job 2 `sums-consistency`. Decision is OUT OF QA scope — user/Gage call. **Not a Quinn blocker** per mission constraints; documented as pending escalation.

### LOW — T4 hook self-check captured as Dev Agent Record evidence, not as separate commit
- **Location:** AC5 sequencing T4 step
- **Issue:** T4 was framed in story as a verification step ("re-run T2 scenario locally with hook installed") rather than a tracked commit. Evidence is inline in Dev Agent Record §Debug Log (bash syntax PASS, COSIGN_REGEX matrix, memory-budget fallback hits = 16). No separate `git log` entry for T4 self-check.
- **Impact:** Auditability is slightly degraded — future reviewers must read Dev Agent Record narrative rather than verify a discrete commit.
- **Mitigation acknowledged:** AC5 step 4 spec says "Hook self-check" — does NOT mandate a separate commit. Inline evidence is conformant with story spec. Riven T7.2 R10 review will validate that C-3 sequencing intent (no `--no-verify` bypass window) is preserved despite no separate T4 commit.
- **Recommendation:** No remediation needed for R15.2; consider tightening spec for future R15-class stories to require T-step-per-commit auditability if desired.

### LOW — `core.hooksPath` is per-clone, not committable
- **Location:** `.githooks/install.sh` lines 24-25, story T3.4 acknowledgment
- **Issue:** Each developer must manually run `bash .githooks/install.sh` after clone. Silent bypass possible if developer forgets.
- **Mitigation:** L2 CI is server-side and non-skippable when T6 lands. L1 is dev-experience optimization, not a security boundary. Acknowledged in install.sh header lines 7-10.
- **Recommendation:** Add post-clone reminder (e.g., README §Setup) suggesting `bash .githooks/install.sh`. Not blocking; defer to follow-up doc PR.
