## 2. Manifest pin decision (CRITICAL — Riven authority)

**Approach chosen: (b) — REMOVE the `PENDING_RIVEN_T8_DUAL_SIGN` placeholder line.**

### Rationale (R15 invariant philosophy)

1. **`state/T002/manifest.json` does NOT exist yet** (verified `ls state/T002/` → only `.gitkeep`). The first canonical run has not occurred. There is no real sha to ratify. Pinning a sentinel string masquerading as a sha is the **exact R15 anti-pattern** the canonical-invariant.sums system exists to prevent.
2. **manifest.json is append-only mutable by design** (story AC9). Even after first canonical run, every subsequent run appends a new line ⇒ the file sha changes by design every run. Pinning content sha would create a permanently-failing invariant or force `cosign-required` on every operational run — neither is sound.
3. **Contract test is satisfied without the placeholder line.** `tests/contracts/test_warmup_orchestrator_contracts.py:144-160` accepts `has_path_entry OR has_schema_marker`. Lines 3-7 of canonical-invariant.sums already contain BOTH `T002.0g-orchestrator` AND `warmup-state-manifest` markers ⇒ `has_schema_marker = True` even after removal.
4. **L2 CI semantics:** `sha256sum -c` (workflow line 76) processes the placeholder line as a malformed checksum. Coreutils tolerance varies; this fragility is itself an R15 risk vector. Removal eliminates the fragility.
5. **Optional Phase 2 (NOT in T8 scope):** if a stable schema-doc invariant is desired, create `docs/architecture/state-T002-manifest-schema.md` and pin its sha (Option (a) extension). This is documentation-engineering, not custodial cosign work, and is recorded as deferred TODO below.
