## §4.1 / §6 Reconciliation Ruling

**The tension.** Spec §4.1 L293 says: "If `target is not None`, call `_guard_canonical_write(target)` once **up-front (fail fast before any DB work)**." A strict reading could argue this means ALWAYS, including dry-run. Spec §6 L357 says: "Changing `--dry-run` semantics — unchanged." Today's dry-run never writes and never connects, so running the guard during dry-run would change semantics (a dry-run with canonical target but unset `VESPERA_MANIFEST_COSIGN` would now raise instead of planning months).

**Dex's ordering.** Guard runs at script L705-706, AFTER the dry-run early-exit at L695-702.

**Ruling: Dex's ordering is correct.** §6 is a scope exclusion ("out of scope") and a binding constraint on behavior preservation. §4.1 "before any DB work" is about **ordering relative to DB connect when writes will occur**, not an override of §6. Treating §4.1 as an override would violate:
- §6 "dry-run semantics unchanged" (explicit spec clause).
- Article IV (No Invention) — reading §4.1 to expand the guard into dry-run invents a semantic not written down.
- Article V (Quality First) — breaking dry-run would force operators to set a valid cosign + sha pin just to plan months, which is the clutter spec Decision 1 rationale (L67-71) explicitly rejects.

Dex defensibly chose to preserve dry-run semantics per §6 and documented the choice in a script comment at L692-694 citing MWF-20260422-1 §6. Ruling: **§6 wins over strict §4.1**.
