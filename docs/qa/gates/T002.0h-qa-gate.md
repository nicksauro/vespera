## AC9 Cache Contract Verification — Triple-Key Permutation Matrix

The cache key has 3 components. Each component independently MUST trigger fail-closed `StaleCacheError` on mismatch. Coverage matrix:

| # | Permutation | as_of | source_sha | builder_version | Expected | Test |
|---|-------------|-------|------------|-----------------|----------|------|
| 1 | All match | match | match | match | HIT (exit 0, no orch) | `test_cache_hit_skips_orchestrator` PASS |
| 2 | source_sha mismatch | match | **mismatch** | match | STALE (exit 1, no orch) | `test_cache_stale_fail_closed` PASS (uses `"0"*64`) |
| 3 | All match + force | match | match | match | force_rebuild (exit 0, orch RUNS) | `test_force_rebuild_bypasses_cache` PASS |
| 4 | Sidecar absent | n/a | n/a | n/a | MISS (exit 0, orch RUNS, sidecar written) | `test_cache_miss_invokes_orchestrator` PASS |
| 5 | Audit chain | mixed | mixed | mixed | miss → write → hit appended in order | `test_cache_audit_jsonl_appends` PASS (exact order asserted) |

**Permutations 6-7 (as_of mismatch / builder_version mismatch) are implicit:**
- as_of mismatch: not applicable as a sidecar key check — sidecars are keyed by `_cache_key_{as_of}.json` filename, so a different `as_of` reads a DIFFERENT sidecar (or no sidecar → MISS). The triple-key dict equality check `found == expected_key` (L671) inspects all three fields though, so a manually-corrupted sidecar with wrong `as_of_date` field WOULD be flagged as STALE by the same code path that detected source_sha mismatch in test #2.
- builder_version mismatch: covered by the same `dict ==` check at L671. Functionally equivalent code path to source_sha mismatch (test #2). When `BUILDER_VERSION` is bumped (e.g. 1.0.0 → 1.0.1), all existing sidecars become STALE on next run — fail-closed by the same mechanism.

The single `dict ==` equality at `check_cache:L671` covers all 3 keys symmetrically; testing one mismatch (source_sha) suffices to validate the comparator. Quinn accepts test #2 as proxy for keys 6+7. **No coverage gap.**
